## RabbitMQ

### 핵심 모델 — 브로커 중심 메시지 라우팅 (AMQP)

```
Producer → Exchange → (Binding 규칙) → Queue → Consumer
```

- 메시지는 Exchange 도착 후, Binding 규칙에 따라 하나 이상의 Queue로 라우팅
- 기본 큐 모델은 **소모형(destructive read)**: 컨슈머가 읽으면 큐에서 제거
- 라우팅 결정을 **브로커가** 담당 → 프로듀서는 Exchange에 던지기만 하면 됨

---

### Exchange 라우팅

#### Exchange 타입

| Exchange 타입 | 동작 | 와일드카드 |
|---|---|---|
| Direct | routing key가 일치하는 큐로 전달 | — |
| Topic | 와일드카드 패턴 매칭 | `*` (단어 1개), `#` (0개 이상) |
| Fanout | 바인딩된 모든 큐에 복사, key 무시 | — |
| Headers | 헤더 속성 기반 라우팅 | — |

**Topic 와일드카드 규칙**
- `*` — 점(`.`)으로 구분된 단어 **정확히 1개** 대체<br />`order.*` → `order.created` ✓, `order.payment.done` ✗
- `#` — 단어 **0개 이상** 대체<br />`order.#` → `order.created` ✓, `order.payment.done` ✓

#### 코드 예시 (Spring AMQP)

#### Dead Letter Exchange (DLX)

```kotlin
@Bean fun deadLetterExchange() = DirectExchange("dlx.exchange")
@Bean fun deadLetterQueue()    = Queue("dlx.queue", true)

@Bean
fun dlxBinding() =
    BindingBuilder
        .bind(deadLetterQueue())
        .to(deadLetterExchange())
        .with("dlx.routing.key")

@Bean
fun orderQueueWithDlx(): Queue =
    QueueBuilder.durable("order.queue")
        .withArgument("x-dead-letter-exchange",    "dlx.exchange")
        .withArgument("x-dead-letter-routing-key", "dlx.routing.key")
        .withArgument("x-message-ttl", 30_000)
        .build()
```

**DLX로 메시지가 이동하는 조건**
1. 컨슈머가 `basicNack` + `requeue=false`
2. 메시지 TTL 만료
3. 큐 길이 초과

<Aside type="tip">
DLX 큐를 별도 컨슈머로 모니터링 + 알림을 설정하면 처리 실패 메시지를 손실 없이 추적할 수 있다.
</Aside>

### 내구성·신뢰성

#### Quorum Queue (RabbitMQ 4.x+ 권장)

```kotlin
@Bean
fun quorumQueue(): Queue =
    QueueBuilder.durable("order.queue")
        .quorum()                            // Raft 기반 복제
        .withArgument("x-delivery-limit", 3) // 최대 재전달 횟수, 초과 시 DLX로 이동
        .build()
```

<Aside type="caution">
RabbitMQ 4.x에서 classic mirrored queue가 제거됐다. 신규 프로젝트는 반드시 **Quorum Queue**를 사용해야 한다.
</Aside>

#### Publisher Confirm — at-least-once 발행 보장

```yaml
spring:
  rabbitmq:
    publisher-confirm-type: correlated
    publisher-returns: true
```

```kotlin
rabbitTemplate.invoke { template ->
    template.convertAndSend("exchange", "routing.key", message)
    template.waitForConfirmsOrDie(5000)
}
```

- 브로커가 큐에 메시지를 저장하면 ack 반환
- `publisher-returns: true` → 라우팅에 실패한 메시지(매칭 큐 없음)를 콜백으로 수신

#### Consumer Ack 전략

```kotlin
@RabbitListener(queues = ["order.queue"], ackMode = "MANUAL")
fun consume(
    message: Message,
    channel: Channel,
    @Header(AmqpHeaders.DELIVERY_TAG) tag: Long
) {
    try {
        processOrder(message)
        channel.basicAck(tag, false)
    } catch (e: Exception) {
        // 실패 → DLX로 이동
        channel.basicNack(tag, false, false)
    }
}
```

| ackMode | 동작 | 주의 |
|---|---|---|
| `AUTO` | 메서드 정상 반환 시 자동 ack | 예외 발생 시 nack → 재큐잉 반복 |
| `MANUAL` | 코드에서 직접 ack/nack 제어 | ack 누락 시 메시지 영구 점유 |
| `NONE` | ack 없음 (at-most-once) | 메시지 유실 가능 |

### Spring 연동 — RabbitMQ

#### Spring Integration vs Spring Cloud Stream

| 항목 | Spring Integration | Spring Cloud Stream |
|---|---|---|
| 추상화 수준 | 낮음 | 높음 (Binder로 브로커 추상화) |
| 코드량 | 많음 (Flow/Adapter/Gateway) | 적음 (함수형 빈) |
| 브로커 교체 | 어댑터 코드 수정 필요 | 의존성 + yml만 변경 |
| Exchange·Queue 선언 | `@Bean` 직접 선언 | 자동 프로비저닝 |
| DLQ 설정 | DLX Bean 직접 선언 | `auto-bind-dlq: true` |
| 학습 비용 | 높음 (EIP 개념 필요) | 낮음 (함수형 모델) |

- 복잡한 메시지 라우팅·변환 파이프라인 → **Spring Integration**
- 빠른 개발, 브로커 교체 가능성, 단순 처리 흐름 → **Spring Cloud Stream**

#### Spring Cloud Stream 핵심 개념

```
애플리케이션 코드 (Function / Consumer / Supplier)
        ↕  Spring Cloud Stream
     Binder (브로커별 구현체)
        ↕
    RabbitMQ / Kafka
```

| 개념 | 설명 |
|---|---|
| **Binder** | 브로커별 구현체. 코드는 브로커를 모름 |
| **Binding** | 함수↔브로커 채널 연결. 이름 규칙: `{함수명}-in-{인덱스}`, `{함수명}-out-{인덱스}` |
| **자동 프로비저닝** | `destination` + `group` 설정만으로 Exchange·Queue 자동 생성 |
| **Consumer Group** | `group: order-service` → 큐 이름: `{destination}.{group}`. 같은 그룹 내 경쟁 소비 |

| 함수 타입 | in | out | 사용 목적 |
|---|:---:|:---:|---|
| `Function<I, O>` | ✓ | ✓ | 수신 후 변환·가공하여 발행 |
| `Consumer<I>` | ✓ | ✗ | 수신 후 처리 (발행 없음) |
| `Supplier<O>` | ✗ | ✓ | 주기적 폴링으로 메시지 발행 |
| `StreamBridge` | ✗ | ✓ | HTTP 요청 등 외부 이벤트 트리거로 즉시 발행 |

```kotlin
@RestController
class OrderController(private val streamBridge: StreamBridge) {

    @PostMapping("/orders")
    fun createOrder(@RequestBody event: OrderEvent): ResponseEntity<Void> {
        val message = MessageBuilder.withPayload(event)
            .setHeader("routingKey", "order.created")
            .build()
        streamBridge.send("order-out", message)
        return ResponseEntity.accepted().build()
    }
}
```

## Kafka

### 핵심 모델 — 분산 append-only 로그

```
Producer → Topic (Partition 0, 1, 2, ...) → Consumer Group
                                              (Offset 기반 위치 추적)
```

- 메시지는 파티션에 **추가(append)** 됨, 소비해도 삭제되지 않음
- 컨슈머는 자신의 **오프셋(offset)** 을 관리 → 과거 메시지 재읽기(replay) 가능
- 라우팅 결정을 **프로듀서(애플리케이션 코드)가** 담당

#### Kafka가 기본으로 보장하는 신뢰성

> 신뢰성 = **"시스템이 예상한 대로 올바르게 작동하는 것"**<br />즉, 보낸 메시지가 **정확히 전달되고, 저장되며, 처리되는 것**

카프카가 기본으로 제공하는 두 가지 보장:

1. **파티션 내 메시지 순서** — 토픽 파티션 내 메시지는 순서대로 저장되고, 컨슈머도 순서대로 처리한다.
2. **레플리카 기반 메시지 보존** — 레플리카가 존재하는 한 메시지는 유실되지 않는다.

이 기본 보장 위에 개발자가 설정을 통해 원하는 수준의 신뢰성을 추가로 달성할 수 있다.

<Aside type="note">
카프카는 `fsync()`로 디스크에 직접 쓰지 않는다. **Page Cache**(커널 메모리)에 먼저 쓰고 이후 디스크로 반영된다. 반영 전에 브로커가 다운되면 유실 가능 → **복제를 통해 다른 브로커의 Page Cache에 데이터를 보관함으로써 이 문제를 해결한다.**

```
MySQL : write → fsync() → 디스크 (브로커 다운돼도 안전)
Kafka : write → Page Cache → (지연 후) 디스크
        → 브로커 다운 시 미반영 데이터 유실 가능
        → 해결: 다른 브로커의 Page Cache에 복제
```
</Aside>

### 파티션과 오프셋

```
Topic: orders
  Partition 0: [msg0] [msg1] [msg4] [msg7]  ← offset 0, 1, 4, 7
  Partition 1: [msg2] [msg3] [msg5]         ← offset 0, 1, 2
  Partition 2: [msg6] [msg8]                ← offset 0, 1

Consumer Group A:
  Consumer 1 → Partition 0 (현재 offset: 5)
  Consumer 2 → Partition 1 (현재 offset: 2)
  Consumer 3 → Partition 2 (현재 offset: 1)
```

- **순서 보장**: 파티션 내에서만 (같은 key → 같은 파티션 → 순서 보장)
- **병렬성**: 파티션 수 = 최대 병렬 컨슈머 수 (컨슈머 > 파티션이면 유휴 컨슈머 발생)
- **재처리**: 오프셋을 과거로 되돌리면 해당 지점부터 재소비 가능

#### 파티션 키 설계

```kotlin
// 좋은 예: entityId를 key로 → 같은 고객의 주문은 항상 같은 파티션
producer.send(ProducerRecord("orders", order.customerId.toString(), orderEvent))

// 나쁜 예: key=null → 라운드로빈 → 같은 고객 주문이 다른 파티션에 분산됨
producer.send(ProducerRecord("orders", null, orderEvent))

// 나쁜 예: 편향된 key → hot partition
producer.send(ProducerRecord("orders", "VIP", orderEvent))
```

**Hot Partition 증상**: 특정 파티션의 lag만 급증, 해당 컨슈머 CPU만 높음

<Aside type="caution">
나중에 파티션 수를 늘리면 key 기반 라우팅이 깨질 수 있다. **초기 파티션 수 설계가 중요하다.**
`처리량 목표 / 단일 파티션 처리량 = 최소 파티션 수`로 계산한다.
</Aside>

#### 컨슈머 그룹

```
Topic: orders (파티션 3개)

Consumer Group A (주문처리팀):  Consumer 1 → P0, P1 / Consumer 2 → P2
Consumer Group B (감사팀):      Consumer 3 → P0, P1, P2  ← 독립적으로 처음부터 소비 가능
```

처리 목적이 다르면 반드시 **별도 컨슈머 그룹**을 설정해야 한다. 같은 그룹에 묶으면 파티션을 나눠 갖기 때문에 각각 전체 메시지가 아닌 일부만 처리하게 된다.

**리밸런스 발생 조건**

1. 컨슈머 추가·제거
2. `max.poll.interval.ms` 초과 (처리 지연)
3. 세션 타임아웃 (`session.timeout.ms`)

<Aside type="caution">
리밸런스 중에는 **처리가 중단**되고, 오프셋 commit이 안 된 메시지는 **중복 처리**될 수 있다.
</Aside>

### 내구성·신뢰성

#### 브로커 설정

| 설정 | 기본값 | 설명 |
|---|---|---|
| `replication.factor` | — | 수동 토픽 생성 시 파티션 복제 수 |
| `default.replication.factor` | — | 자동 토픽 생성 시 파티션 복제 수 |
| `min.insync.replicas` | 1 | `acks=all` 시 최소 확인 복제 수 |
| `log.retention.hours` | 168 (1주일) | 메시지 보관 기간 |
| `log.retention.bytes` | -1 (무제한) | 보관 크기 제한. `retention.hours`와 동시 사용 시 혼란 → 하나만 권장 |
| `log.segment.bytes` | 1 GB | 세그먼트 최대 크기 |
| `message.max.bytes` | 1 MB | 브로커 허용 최대 메시지 크기 |

<Aside type="caution">
**`log.segment.bytes` 주의**: 하루 100 MB씩 쌓이고 `log.segment.bytes=1 GB`(기본값), `log.retention.hours=168`(1주일)이면 세그먼트가 닫히지 않아 **최대 17일**까지 메시지가 보관될 수 있다.
</Aside>

<Aside type="note">
**복제 수 설계**: 브로커 노드 수에 맞춰 설정한다.
예) 3 브로커 → `replication.factor=3`, `min.insync.replicas=2`

`min.insync.replicas`를 `replication.factor`와 동일하게 설정하면 가용성이 떨어지므로 주의한다. `message.max.bytes`는 컨슈머의 `fetch.message.max.bytes`와 맞춰야 한다 — 브로커 값이 더 크면 컨슈머가 메시지를 읽지 못하는 상황이 발생한다.
</Aside>

#### 프로듀서 설정

```yaml
spring:
  kafka:
    producer:
      acks: all                    # min.insync.replicas 수만큼의 복제 파티션에 기록될 때까지 대기
      retries: 3
      enable-idempotence: true     # 중복 발행 방지 (acks=all 필요)
      properties:
        delivery.timeout.ms: 120000   # 재시도는 이 시간 범위 내에서만 수행 (기본 2분)
        request.timeout.ms: 30000
        max.in.flight.requests.per.connection: 5
```

| `acks` 설정 | 의미 | 내구성 | 처리량 |
|---|---|:---:|:---:|
| `0` | 브로커 응답 없이 발행 | 낮음 | 최고 |
| `1` | 리더만 확인 | 중간 | 높음 |
| `all` | `min.insync.replicas`만큼 복제 확인 | 높음 | 낮음 |

**동기 vs 비동기 발행**

```kotlin
// 동기 — 처리량 낮음
val future = kafkaTemplate.send("orders", orderEvent)
future.get()

// 비동기 — 권장
kafkaTemplate.send("orders", orderEvent)
    .whenComplete { result, ex ->
        if (ex != null) log.error("전송 실패", ex)
        else log.info("전송 성공: offset=${result.recordMetadata.offset()}")
    }
```

**max.in.flight.requests.per.connection과 순서 역전**

프로듀서는 설정값만큼 메시지를 동시에 전송하며, 재시도 시 순서가 뒤바뀔 수 있다.

```
[msg1, msg2, msg3] 동시 전송
→ msg1 실패 → 재시도
→ msg2, msg3 먼저 도착
→ 결과: [msg2, msg3, msg1]  ← 순서 역전
```

`enable.idempotence=true` (멱등성 프로듀서)로 해결 → 브로커가 **Producer Id + Sequence Number**로 순서를 보장하고 중복을 제거한다.

<Aside type="caution">
**애플리케이션 레벨에서 재시도 로직을 별도로 구현하면 안 된다.** 프로듀서 내부에 재시도 메커니즘이 내장되어 있으므로, 추가 재시도 시 `delivery.timeout.ms`(기본 2분) 범위 안에서 메시지 중복이 발생할 수 있다.
</Aside>

**멱등성 프로듀서의 한계**

- 프로듀서 앱 **크래시 후 재부팅** → 새 Producer Id 할당 가능 → 이전 메시지를 모르고 재전송 → **중복 발생 가능**
- **다중 프로듀서** 환경: 동일 파티션에 경쟁적으로 전송 시 전역 순서 보장 불가

**트랜잭셔널 프로듀서**

멱등성 프로듀서의 한계를 넘으려면 트랜잭셔널 프로듀서로 업그레이드한다.

```yaml
spring:
  kafka:
    producer:
      transaction-id-prefix: tx-order-  # 고정 transactional.id → 재부팅 후에도 동일 Producer Id
    consumer:
      isolation-level: read_committed
```

```kotlin
// 둘 다 성공하거나 둘 다 실패
kafkaTemplate.executeInTransaction { t ->
    t.send("orders", orderEvent)
    t.send("inventory", inventoryEvent)
}
```

- 재부팅 후에도 **동일 Producer Id** 유지
- **Epoch Number**로 이전 인스턴스(좀비 프로듀서)를 무효화 → 중복 방지

#### 컨슈머 설정

```yaml
spring:
  kafka:
    consumer:
      auto-offset-reset: earliest
      enable-auto-commit: false
      max-poll-records: 500
      properties:
        max.poll.interval.ms: 300000
        session.timeout.ms: 45000
        isolation.level: read_committed
```

**auto.offset.reset**

`auto.offset.reset`은 오프셋이 없거나 만료된 경우 소비를 시작할 위치를 결정한다.
오프셋은 `consumer_offsets` 토픽에 저장되며, 활성 컨슈머가 없는 상태가 `offset.retention.minutes`(기본 7일) 이상 지속되면 삭제될 수 있다.

| 값 | 동작 | 사용 상황 |
|---|---|---|
| `earliest` | 파티션 첫 메시지부터 읽기 | 메시지 누락 방지 (중복 처리 감수) |
| `latest` | 파티션 마지막 메시지부터 읽기 | 최신 메시지만 필요한 경우 |
| `none` | 오프셋 없으면 에러 발생 후 거부 | 오프셋 상태를 엄격히 관리하는 경우 |

**오프셋 커밋 — 자동 vs 수동**

| 방식 | 특징 |
|---|---|
| **자동** `enable.auto.commit=true` | 다음 `poll()` 호출 시 커밋. 안전하지만 타이밍 세밀 제어 불가 |
| **수동** `enable.auto.commit=false` | 처리 완료 후 명시적 커밋. 시점 제어 가능, 중복 처리 예방 |

<Aside type="note">
자동 커밋은 **메시지 처리 후** 다음 `poll()` 호출 시점에 커밋되므로 처리 자체는 안전하다. 다만 커밋 타이밍을 세밀하게 제어하기 어려워 중복 처리가 발생할 수 있다. 수동 커밋은 커밋 시점을 정확히 제어할 수 있어 중복 처리를 줄일 수 있다.
</Aside>

**Dead Letter Queue (DLQ)**

```
메시지 소비 → 처리 실패 → 재시도
                        → 재시도 반복 실패
                        → DLQ 토픽으로 분리
                           ↓
                        모니터링·알림 + 추후 처리 재개
```

```yaml
spring:
  cloud:
    stream:
      kafka:
        bindings:
          consumeOrder-in-0:
            consumer:
              enable-dlq: true
              dlq-name: orders.dlq
              dlq-producer-properties:
                retries: 0
```

무한 재시도는 시스템 부담을 키우고 전체 처리 성능을 저하시킨다. DLQ로 분리해 정상 처리 흐름을 유지하면서 실패 메시지를 유실 없이 보존한다.

**컨슈머 리밸런스·지연 관리**

```
poll() 호출 → 처리 시작
300초 경과 (max.poll.interval.ms 기본값) → 브로커가 컨슈머 dead로 간주
→ 리밸런스 발생 → 처리 중이던 파티션 다른 컨슈머에게 재할당
→ 처리 완료 전에 재처리 발생 (중복)
```

튜닝 방향: `max.poll.records` 줄이기 / `max.poll.interval.ms` 늘리기 / 오래 걸리는 작업은 비동기 처리 후 수동 commit

### Exactly-Once Semantics (EOS)

```
Kafka 내부 경로만 exactly-once:
  Producer → Kafka Broker → Consumer (오프셋 commit)

외부 부작용은 별도 처리 필요:
  Consumer → DB write   ← exactly-once 보장 안 됨
  Consumer → HTTP call  ← exactly-once 보장 안 됨
```

**Outbox 패턴 — 외부 exactly-once**

```
1. DB 트랜잭션 내에서 이벤트를 outbox 테이블에 저장
2. Outbox 테이블을 폴링(또는 CDC)해서 Kafka로 발행
3. 컨슈머는 idempotency key로 중복 처리 방지
```

```kotlin
@KafkaListener(topics = ["orders"])
fun consume(event: OrderEvent) {
    // 이미 처리된 이벤트
    if (processedEventRepository.exists(event.eventId)) return

    // 같은 DB 트랜잭션 내에서
    orderRepository.save(order)
    processedEventRepository.save(event.eventId)
}
```

**Read-Process-Produce 커밋 타이밍**

```
올바른 순서:
  1. 메시지 소비 (Read)
  2. 비즈니스 처리 (Process)
  3. 새 토픽에 메시지 발행 성공 (Produce)
  4. 오프셋 커밋  ← 반드시 마지막

또는: 트랜잭셔널 프로듀서로 커밋 + 발행을 원자적으로 처리
```

| 기능 | 설정 | 효과 |
|---|---|---|
| Idempotent producer | `enable.idempotence=true` | 네트워크 재시도로 인한 중복 발행 방지 |
| 트랜잭션 | `transactional.id` 설정 | 여러 토픽에 원자적 발행 |
| `read_committed` | `isolation.level` 설정 | 커밋된 메시지만 소비 |
| Kafka Streams EOS | `processing.guarantee=exactly_once_v2` | 스트림 처리 내 exactly-once |

## RabbitMQ vs Kafka 비교

### 핵심 모델 차이

| 항목 | RabbitMQ | Kafka |
|---|---|---|
| 기본 모델 | 브로커 중심 큐·라우팅, 소비 시 제거 | 분산 append-only 로그, 오프셋 기반 재읽기 |
| 라우팅 책임 | Exchange Binding으로 브로커가 결정 | 프로듀서가 토픽·파티션을 직접 결정 |
| 강점 | 라우팅 유연성, RPC·작업 큐, 낮은 운영 진입장벽 | 고처리량, fan-out, 리플레이·백필, 스트리밍 파이프라인 |
| 내구성·복제 | Quorum Queue(Raft) + publisher confirm/ack | 복제+ISR, `acks=all` + `min.insync.replicas` |
| 순서 보장 | 큐 단위 FIFO (병렬 소비·재큐잉 시 흔들릴 수 있음) | **파티션 내** 순서만 강하게 보장 |
| 재처리·이력 | 기본 큐는 소모형, Streams로 리플레이 지원 | 로그 보관·압축으로 재처리·상태 복구에 유리 |
| 운영 복잡도 | 중소규모 워크로드에 비교적 단순 | 파티션·리밸런스·스토리지·스키마 관리 필요 |
| 최신 버전 | 4.x에서 classic mirrored queue 제거, quorum/streams 권장 | 4.0+에서 ZooKeeper 제거, KRaft 전용 |

### 라우팅 책임 비교

```
# RabbitMQ: Binding 설정만으로 완결 (프로듀서 코드 무변경)
"log.error" → error-alert-queue
"log.warn"  → warn-queue
"log.*"     → all-log-queue (동시 라우팅)

# Kafka: 아래 중 하나를 선택해야 함
1. 컨슈머가 메시지 읽고 직접 분기 처리
2. 컨슈머가 다른 토픽에 재발행
3. Kafka Streams 라우팅 파이프라인 별도 구성
```

```kotlin
// Kafka에서 조건 분기 — 컨슈머가 직접 처리해야 함
// 신규 구독자 추가 → 컨슈머 코드 수정 필요
@KafkaListener(topics = ["logs"])
fun consume(log: LogEvent) {
    when (log.level) {
        "ERROR" -> alertService.send(log)
        "WARN"  -> monitorService.record(log)
    }
    storageService.save(log)
}
```

<Aside type="note">
단순 pub/sub 패턴에서는 두 시스템 차이가 크게 체감되지 않는다. 조건부 선택적 라우팅이나 신규 구독자 추가가 잦은 환경에서 RabbitMQ의 브로커 중심 설계가 유리하다.
</Aside>

### 언제 무엇을 선택할까

| 상황 | 선택 | 이유 |
|---|---|---|
| 작업 큐, RPC, 복잡한 라우팅 | RabbitMQ | Exchange Binding으로 라우팅 유연성 확보, 낮은 운영 복잡도 |
| 서비스 간 커맨드·응답, 짧은 메시지 수명 | RabbitMQ | 소모형 큐 모델이 적합 |
| 이벤트 스트리밍, 재처리·감사 추적 | Kafka | 로그 보관 + 오프셋 재처리 |
| 다수 컨슈머 fan-out, 데이터 파이프라인·CDC | Kafka | 독립적 컨슈머 그룹 + 고처리량 |
| 트랜잭셔널 커맨드 + 이벤트 백본 병행 | 혼합 | 커맨드는 RabbitMQ, 이벤트·분석은 Kafka |

```
혼합 전략 예시
  [서비스 A] ──커맨드(RabbitMQ)──▶ [서비스 B]
  [서비스 B] ──이벤트(Kafka)────▶  분석 파이프라인 / 알림 서비스 / 감사 로그
```

### Kafka의 주요 이점

**재처리 가능 아키텍처**

오프셋을 과거로 되돌려 특정 시점부터 재소비할 수 있다. 버그 수정 후 이미 처리한 메시지를 다시 처리해야 하는 상황에서 핵심적인 강점이다.

```bash
kafka-consumer-groups.sh --reset-offsets \
    --group order-service \
    --topic orders \
    --to-datetime 2026-03-08T00:00:00.000
# 재시작 → 해당 시점부터 재소비
```

- `cleanup.policy=compact` → 같은 key의 최신 값만 보존 → 최신 상태 스냅샷 + 변경 이력 동시 구현

**느슨한 결합 — 다수 다운스트림**

```
orders 토픽:
  Consumer Group: order-service        (주문 처리)
  Consumer Group: inventory-service    (재고 차감)
  Consumer Group: notification-service (알림)
  Consumer Group: analytics-service    (분석) ← 신규 추가 시 프로듀서 코드 무변경
```

파티션 수 = 병렬 컨슈머 수 → 처리량 선형 확장. 대용량 파이프라인(로그·CDC·분석 적재)에 적합하다.

### Kafka 도입 시 주의할 점

**1. Exactly-once 오해**
Kafka EOS는 Kafka 내부 경로에만 적용된다. 외부 DB·HTTP 부작용은 자동으로 exactly-once가 되지 않는다.
→ `Outbox + idempotency key + upsert` 조합으로 처리한다.

**2. 파티션 키 설계 실패**
잘못된 key → hot partition → 처리량 저하, 순서 보장 실패.
- 엔터티 단위 순서 필요 → entityId를 key로
- 순서 불필요 → null key (라운드로빈) 또는 고른 분포의 key

**3. 스키마 거버넌스 부재**

```
v1 이벤트: { "orderId": 1, "amount": 1000 }
v2 이벤트: { "orderId": 1, "totalAmount": 1000 }  ← 필드명 변경

→ v1 컨슈머가 v2 이벤트 수신 → 파싱 실패 → 컨슈머 장애
```

Schema Registry(Confluent 또는 AWS Glue)를 사용하고, Backward 호환(신규 필드 추가만)을 원칙으로 한다.

**4. 운영 난이도 과소평가**

| 항목 | 내용 |
|---|---|
| 브로커 수 | 최소 3개 (과반수 장애 허용) |
| replication factor | 3 권장 (`min.insync.replicas=2`) |
| 저장소 IOPS | 파티션 수 × 쓰기 처리량 기준 |
| retention | 기간(`retention.ms`) vs 크기(`retention.bytes`) — 하나만 |
| 모니터링 지표 | consumer lag, ISR 수, under-replicated 파티션 |

<Aside type="caution">
Kafka 4.0+는 KRaft 전용이다. 기존 ZooKeeper 기반 운영 중이면 마이그레이션 계획을 반드시 수립해야 한다.
</Aside>

**5. 모니터링을 lag만 보는 실수**

```
records-lag-max만 보면 놓치는 것들:
  fetch-latency             → 브로커 응답 지연 (네트워크·디스크 병목)
  throttle-time             → 쿼터 초과로 인한 속도 제한
  rebalance 빈도            → 컨슈머 불안정 신호
  ISR 수                    → 복제 지연 or 브로커 장애 신호
  under-replicated-partitions → 데이터 유실 위험
```
